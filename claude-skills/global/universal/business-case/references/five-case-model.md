# Five Case Model -- Full Business Case Template

The Five Case Model (HM Treasury Green Book standard) provides the backbone for major business cases. Each case answers a distinct question. Together they give decision-makers a complete picture.

## 1. Strategic Case

**Question answered:** Why should we do this? How does it connect to strategy?

### Sections

#### 1.1 Problem or Opportunity Statement
- What is the problem, gap, or opportunity?
- Who is affected and how?
- What evidence supports the existence of this problem? (data, trends, incidents, customer feedback)
- What happens if we do nothing? (the "burning platform" -- quantify the cost of inaction)

#### 1.2 Strategic Alignment
- Which organizational strategy, OKR, or priority does this serve?
- How does it advance the mission?
- If there's a technology strategy or product roadmap, where does this fit?

#### 1.3 Why Now
- What has changed to make this urgent or timely?
- Regulatory deadline, competitive move, technology maturity, market window
- Cost of delay (quantified if possible)

#### 1.4 Scope
- What is in scope and what is explicitly out of scope
- Key constraints (budget, timeline, regulatory, technical, organizational)
- Dependencies on other projects or decisions

### Writing guidance for Strategic Case
Lead with impact, not background. The reader should understand the stakes in the first paragraph. "We will lose X market share" is stronger than "The market is evolving." Strategic alignment should be specific -- name the OKR, the strategy pillar, the board directive. Vague references to "digital transformation" don't help.

## 2. Economic Case

**Question answered:** What are the options, and which gives best value?

### Sections

#### 2.1 Options Analysis
Present 3-5 options including:
- **Option 0: Do nothing** (always include -- this is the baseline)
- **Option 1: Minimum viable** (lowest cost that addresses the core problem)
- **Option 2: Recommended** (best balance of value, cost, and risk)
- **Option 3: Maximum scope** (full solution, higher cost)
- **Option 4: Alternative approach** (buy vs. build, partner vs. internal, etc.)

For each option, provide:
| Criterion | Option 0 | Option 1 | Option 2 | Option 3 |
|-----------|----------|----------|----------|----------|
| Estimated cost | | | | |
| Expected benefits | | | | |
| Key risks | | | | |
| Time to value | | | | |
| Strategic fit | | | | |

#### 2.2 Cost-Benefit Analysis
For the recommended option:
- **Costs**: Capital (one-time), operational (recurring), opportunity cost, transition costs
- **Benefits**: Revenue increase, cost savings, risk reduction, capability building, strategic optionality
- Quantify in monetary terms where possible
- For non-monetary benefits, use a scoring scale with clear definitions

#### 2.3 Sensitivity Analysis
- What if costs are 20% higher? 50% higher?
- What if benefits are 20% lower? 50% lower?
- What if the timeline slips by 6 months?
- Identify the break-even point: at what benefit level does the investment stop making sense?

#### 2.4 Recommendation
- State the recommended option clearly
- Explain why it beats the alternatives
- Acknowledge what you're giving up compared to higher-scope options

### Writing guidance for Economic Case
The options analysis is the heart of the business case. If only one option is presented, the case is a justification, not an analysis. "Do nothing" must be a real option with real consequences, not a straw man. Sensitivity analysis should focus on the assumptions the decision hinges on, not every variable.

## 3. Commercial Case

**Question answered:** How will this be delivered or procured?

### Sections

#### 3.1 Delivery Approach
- Build in-house, buy off-the-shelf, contract, partnership, hybrid
- Rationale for the chosen approach

#### 3.2 Procurement Strategy (if applicable)
- Vendor landscape (who can deliver this?)
- Selection criteria and weighting
- Contract structure (fixed-price, T&M, outcome-based)
- IP ownership and licensing considerations

#### 3.3 Key Partnerships and Dependencies
- External partners, vendors, or collaborators
- What they bring and what we bring
- Contractual or relationship risks

#### 3.4 Make vs. Buy Analysis (if relevant)
- Total cost of ownership for each approach (5-year view)
- Capability implications (what do we learn/build vs. outsource)
- Control and flexibility implications
- Vendor lock-in risk

### Writing guidance for Commercial Case
This section is often the weakest in internal business cases because the author assumes "we'll just build it ourselves." If that's the recommendation, justify it against alternatives. Make vs. buy decisions should include hidden costs: maintenance burden, talent acquisition, opportunity cost of internal teams.

## 4. Financial Case

**Question answered:** What does it cost, and can we afford it?

### Sections

#### 4.1 Cost Breakdown
| Cost Category | Year 1 | Year 2 | Year 3 | Year 4 | Year 5 | Total |
|--------------|--------|--------|--------|--------|--------|-------|
| Personnel | | | | | | |
| Equipment/licenses | | | | | | |
| External services | | | | | | |
| Infrastructure | | | | | | |
| Training/change mgmt | | | | | | |
| Contingency (10-20%) | | | | | | |
| **Total** | | | | | | |

#### 4.2 Funding Source
- Which budget does this come from?
- Is funding secured, requested, or contingent?
- Multi-year funding implications

#### 4.3 Financial Metrics
- **NPV** (Net Present Value): Use organization's standard discount rate. State the rate explicitly.
- **IRR** (Internal Rate of Return): What return does this investment generate?
- **Payback period**: When does the investment break even?
- **ROI**: (Net benefit / Cost) x 100

#### 4.4 Cash Flow Projection
Show year-by-year: costs out, benefits in, cumulative position.

#### 4.5 Financial Risks
- What financial assumptions are most uncertain?
- What triggers would require additional funding?
- What is the maximum downside exposure?

### Writing guidance for Financial Case
Be conservative in benefit projections, realistic in cost estimates. Include contingency (10-20% depending on uncertainty). If you can't quantify a benefit, don't force a number -- state it qualitatively and note that it's excluded from the financial metrics. Decision-makers respect honesty about uncertainty more than false precision.

## 5. Management Case

**Question answered:** Can we actually deliver this, and how will we govern it?

### Sections

#### 5.1 Governance Structure
- Project sponsor (name and role)
- Steering committee composition
- Decision-making authority and escalation path
- Reporting cadence and format

#### 5.2 Project Plan Summary
- Key phases and milestones
- Critical path
- Resource plan (team composition, key roles)
- Start date, key decision points, expected completion

#### 5.3 Risk Register

| Risk | Likelihood | Impact | Mitigation | Owner |
|------|-----------|--------|------------|-------|
| | H/M/L | H/M/L | | |

Include at least:
- Top 3 technical risks
- Top 2 organizational/people risks
- Top 1 external/market risk

#### 5.4 Success Criteria and KPIs
- What measurable outcomes define success?
- When will they be measured?
- Who is accountable for each?

#### 5.5 Change Management
- Who is affected by this change?
- What changes to processes, tools, or roles?
- Communication plan
- Training requirements

#### 5.6 Dependencies and Constraints
- What must happen before this project can start?
- What other projects depend on this one?
- Resource contention with other priorities

### Writing guidance for Management Case
This is where business cases often fail in execution. A brilliant Strategic Case means nothing if the Management Case is hand-waving. Name specific people, not roles. "Led by a senior engineer" is weaker than "Led by [Name], who delivered [similar project] in 2024." The risk register should include things that might actually happen, not generic placeholders.

## Writing Guidance

### General Principles

1. **Executive summary first**: Write a 1-page executive summary after completing all 5 cases. It should stand alone -- a busy executive should get the full picture from this page.
2. **Assumptions register**: Maintain a numbered list of key assumptions referenced throughout. This lets reviewers challenge assumptions without reading the whole document.
3. **Version control**: Include version number, date, author, status (Draft/Review/Approved) on the cover.
4. **Appendices**: Move detailed calculations, market research data, and technical specifications to appendices. The main document should flow as a narrative.

### Common Pitfalls

- **Advocacy masquerading as analysis**: The business case should enable a decision, not sell a predetermined conclusion. If the "do nothing" option is dismissed in one sentence, the case has a credibility problem.
- **Missing opportunity cost**: Every resource allocated here can't be used elsewhere. Name what you're displacing.
- **Benefit double-counting**: Ensure the same benefit isn't claimed by multiple projects in the portfolio.
- **Optimism bias**: Research shows project costs are underestimated by 20-50% on average. Acknowledge this and include appropriate contingency.
- **Ignoring the political landscape**: A technically sound case can fail if it doesn't account for organizational dynamics. The Management Case should address this.
